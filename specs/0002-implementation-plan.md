# 0002 实现计划

## 1. 文档信息

- 文档编号：`0002`
- 文档名称：视频字幕提取与翻译软件实现计划
- 上游规格：`specs/0001-spec.md`
- 目标：将 `0001` 中已确定的产品与技术约束拆解为可执行的开发计划

## 2. 实现目标

首版实现以下闭环能力：

- 基于 `Rust + Tauri 2` 的桌面应用框架
- 导入单个/多个视频文件并形成任务列表
- 选择输出目录，默认输出到视频同目录
- Whisper 模型下载、管理、模型目录落盘
- 音频抽取、转写、生成 `srt`
- 可关闭翻译，或使用 OpenAI 兼容 LLM 进行翻译
- 三种字幕生成模式
- 任务开始、软暂停、删除、清空
- GPU/CPU 检测、Whisper GPU 开关、推荐模型档位
- 术语表、翻译批次、翻译失败回退
- 严格便携模式的数据/配置/密钥存储

## 3. 总体实施策略

### 3.1 开发原则

- 先打通单任务完整链路，再扩展为批量与并发
- 先保证 `关闭翻译 + LLM 翻译` 两条正式路径稳定，再考虑实验性翻译引擎
- 先实现任务执行可靠性，再补性能优化
- 先统一数据模型与任务快照机制，再做 UI 联动
- 先交付最小可用桌面版，再补高级体验

### 3.2 范围边界

本计划只覆盖 `0001` 已确认的首版正式范围。

不纳入本计划主线：

- Google Web 免费模式的稳定交付
- 字幕编辑器
- 多字幕格式
- 自动恢复未完成队列
- 多 Whisper 后端并行支持

## 4. 建议架构

### 4.1 技术栈

- 桌面容器：`Tauri 2`
- 核心语言：`Rust`
- 前端：建议 `React + TypeScript`
- 异步运行时：`tokio`
- 序列化：`serde`
- 配置格式：`json`
- 任务状态持久化：首版可用 `json`，后续可迁移 `sqlite`
- 音频抽取：`ffmpeg`

   - 任务编排

## 6. 数据模型与状态设计

### 6.1 配置模型

至少包含：

- `translator.engine`
- `llm.base_url`
- `llm.model`
- `whisper.model`
- `whisper.use_gpu`
- `whisper.recognition_lang`
- `translate.source_lang`
- `translate.target_lang`
- `translate.style`
- `subtitle.mode`
- `subtitle.output_dir_mode`
- `subtitle.custom_output_dir`
- `runtime.max_parallel_tasks`
- `runtime.task_auto_retry_max`

### 6.2 任务模型

至少包含：

- `id`
- `video_path`
- `file_name`
- `status`
- `progress`
- `phase`
- `translator_engine`
- `config_snapshot_id`
- `config_snapshot_summary`
- `output_paths`
- `error_message`
- `created_at`
- `updated_at`

### 6.3 状态流转

建议任务状态流转如下：

`待开始 -> 排队中 -> 提取音频中 -> 识别中 -> 翻译中 -> 导出中 -> 已完成`

异常分支：

- 任意阶段 -> `暂停请求中` -> `已暂停`
- 任意阶段 -> `失败`
- `失败` -> `开始` -> 重新执行
- `已暂停` -> `开始` -> 继续执行

### 6.4 配置快照策略

任务在点击开始时生成快照，并写入任务记录。

快照必须至少覆盖：

- 翻译引擎
- 字幕模式
- Whisper 模型
- 是否使用 GPU
- 识别语言
- 翻译源语言/目标语言
- 输出目录
- 输出覆盖规则
- 术语表版本
- 翻译风格
- 最大重试次数

## 10. 风险与应对

### 10.1 Whisper 后端集成风险

风险：

- GPU 支持与推理库兼容性不稳定

应对：

- 首版仅选一个实际可落地的 Whisper 后端
- CPU 路径必须可用
- GPU 不可用时自动降级或明确提示

### 10.2 LLM 接口兼容风险

风险：

- 各家 OpenAI 兼容实现细节不同

应对：

- 统一抽象请求结构
- 保持适配层隔离
- 连接测试返回清晰错误

### 10.3 批量翻译错位风险

风险：

- LLM 返回段落不一致导致字幕错位

应对：

- 严格编号校验
- 失败重试
- 自动缩批
- 最终逐条回退

### 10.4 便携模式安全风险

风险：

- 密钥虽然加密，但仍和软件放在同目录

应对：

- 明确 UI 提示边界
- 不记录明文 Key
- 加密文件和配置文件分离

### 10.5 Windows 目录权限风险

风险：

- 用户将软件放到不可写目录

应对：

- 启动时检测软件目录写权限
- 无权限时阻止关键流程并明确提示

## 14. 增量实现计划：翻译引擎扩展与原字幕智能分段

本节对应 `0001` 新增设计，重点补足两块能力：

- `翻译引擎` 明确支持 `关闭 / LLM / Google Web 免费模式（实验性）`
- `原字幕分段策略` 明确支持 `关闭 / 自动 / 规则优先 / LLM 优先`

实现原则：

- 翻译和原字幕分段必须解耦，不能共用一个开关
- 原字幕分段发生在 `Whisper 转写之后、翻译之前`
- `Google Web 免费模式（实验性）` 只负责翻译，不参与原字幕分段
- `自动` 分段模式在无可用 LLM 配置时必须自动回退到规则分段
- 首版原字幕分段目标是“观感基本对齐”，不是独立 `forced alignment`

### 14.1 增量目标

本轮新增实现目标：

- 用户可独立配置翻译引擎，不影响原字幕分段策略
- 用户可在不启用翻译时，单独对原字幕做智能分段
- 系统可根据配置在 `LLM 分段` 和 `规则分段` 之间自动切换或回退
- 原字幕分段后的时间轴优先使用 `Whisper` 词级时间；无词级时间时退化为近似分配
- 翻译阶段以“分段后的原字幕”作为输入，而不是以 Whisper 原始片段直接翻译
- 任务快照需固化 `分段策略 + 时间策略 + 阈值`

### 14.2 配置模型增量

需在配置模型中新增独立分段配置组，例如：

```json
{
  "segmentation": {
    "strategy": "auto",
    "max_chars_per_segment": 42,
    "max_duration_seconds": 6.0,
    "timing_mode": "word_timestamps_first"
  }
}
```

新增字段说明：

- `segmentation.strategy`
  - `disabled`
  - `auto`
  - `rules_only`
  - `llm_preferred`
- `segmentation.max_chars_per_segment`

## 15. 增量实现计划：`whisper.cpp + VAD` 转写链路

### 15.1 增量背景

`0001` 新增需求已经明确：

- 首版转写后端固定为 `whisper.cpp + VAD`
- 不在首版切换到 `faster-whisper` 或 Python 服务后端
- 目标优先解决“片头音乐、长静音、明显非语音导致字幕从 0 秒开始或时间轴提前”的问题

因此本节实现计划聚焦于：

- 在现有 `whisper.cpp` 路线中补齐 `VAD` 预处理
- 保持当前 Rust + Tauri + 本地 CLI 架构不变
- 尽量复用现有音频抽取、任务调度、字幕分段与导出链路

### 15.2 增量目标

- 用户可在配置页启用或关闭 `VAD`
- 系统可在转写前先识别语音区间，再调用 `whisper.cpp`
- 常见“前几秒纯音乐、后面才开口”的视频中，首条字幕起点明显更接近真实说话时间
- `VAD` 输出的语音片段可正确映射回原始时间轴
- 现有原字幕分段能力继续工作，并建立在 `VAD + whisper.cpp` 的结果之上
- 当 `VAD` 关闭时，系统仍可按旧路径转写
- 当 `VAD` 执行失败时，系统给出明确错误，不静默伪装成受保护的 `VAD` 路径

### 15.3 范围边界

本节包含：

- `VAD` 配置项
- `VAD` 运行时依赖管理
- 音频切段、时间偏移映射
- `whisper.cpp` 多片段转写整合
- 任务状态与日志补充
- 测试与验收补充

本节不包含：

- 切换到 `faster-whisper`
- 引入 Python 环境作为首版主依赖
- 独立 `forced alignment`
- 多 ASR 后端并行支持

### 15.4 推荐技术方案

建议采用“前置 VAD + 片段转写 + 结果回拼”的主链路：

1. `ffmpeg` 抽取单声道 16k wav
2. `VAD` 读取 wav，识别语音区间
3. 对语音区间按最大时长进一步裁切
4. 将每个语音区间分别送给 `whisper.cpp`
5. 把每段转写结果按区间偏移量映射回全局时间轴
6. 合并为统一内部 `cue` 结构
7. 继续进入原字幕分段、翻译和导出

推荐原因：

- 不要求替换现有 `whisper.cpp` 后端
- 对当前架构改动集中在“识别前后处理层”
- 更容易控制“片头音乐”和“长静音”场景
- 后续若要做更强对齐，可继续叠加，不会推翻本层设计

### 15.5 配置模型增量

需在 `whisper` 配置组中新增：

约束规则：

- `enable_vad`

- `vad_threshold`

- `vad_min_speech_ms`
  - 合法范围建议 `100 ~ 5000`

  - 合法范围建议 `50 ~ 3000`
- `vad_max_segment_ms`
  - 合法范围建议 `3000 ~ 30000`

### 15.13 风险与应对

#### A. `VAD` 误切风险

- 可能把短促人声误判为非语音，或把音乐边缘误判为语音
- 应对：
  - 配置阈值可调
  - 保留关闭 `VAD` 的开关
  - 通过“最小语音时长 / 最小静音时长”做平衡

#### B. 切片过碎风险

- 片段太碎会增加 `whisper.cpp` 调用次数和拼接复杂度
- 应对：
  - 相邻近距离区间合并
  - 控制最小片段长度
  - 增加单段最大语音时长阈值

#### C. 跨片段拼接风险

- 邻近片段可能产生时间重叠、文本断裂或排序异常
- 应对：
  - 严格按全局时间排序
  - 合并前统一做时间偏移
  - 增加边界去重和重叠修正

#### D. 运行时依赖风险

- `VAD` 资源可能下载失败、缺失或初始化失败
- 应对：
  - 将运行时资源纳入软件目录统一管理
  - 检测失败时给出明确错误
  - 不伪装成“已启用 VAD”

### 15.14 本节交付定义

当以下条件满足时，可视为 `0001` 中 `whisper.cpp + VAD` 新增需求已完成对应实现：

- 配置中可独立开启或关闭 `VAD`
- `VAD` 配置项可保存、可快照、可生效
- 转写主流程已变为 `ffmpeg -> VAD -> whisper.cpp -> 原字幕分段 -> 翻译/导出`
- 常见片头音乐/静音场景下，首条字幕起点明显优于“整段音频直接转写”
- `VAD` 关闭时旧链路仍可正常工作
- `VAD` 失败时系统给出明确错误，不静默冒充成功
- 前端配置页、任务进度、测试计划和验收条目均覆盖 `VAD`
